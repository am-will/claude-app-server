# Protocol v1 — Canonical JSON-RPC Contract

Status: Draft (implementation-ready)

## 1) Scope and Design Goals

This document defines the provider-agnostic wire contract for `claude-app-server` using JSON-RPC 2.0 framing.

Design goals:
- Canonical outbound shape is **camelCase**.
- Inbound requests may use camelCase or snake_case for compatibility.
- Transport-agnostic envelope (stdio, WebSocket, HTTP streaming, etc.).
- Deterministic errors and capability negotiation.

Normative terms **MUST**, **SHOULD**, **MAY** are used as in RFC 2119.

---

## 2) Core Framing

All request/response messages follow JSON-RPC 2.0:

```json
{
  "jsonrpc": "2.0",
  "id": "req-123",
  "method": "session.start",
  "params": {}
}
```

Responses:

```json
{
  "jsonrpc": "2.0",
  "id": "req-123",
  "result": {}
}
```

Errors:

```json
{
  "jsonrpc": "2.0",
  "id": "req-123",
  "error": {
    "code": 1001,
    "message": "Invalid params",
    "data": {
      "path": "params.maxTokens",
      "reason": "must be >= 1"
    }
  }
}
```

Notifications (no `id`):

```json
{
  "jsonrpc": "2.0",
  "method": "conversation.delta",
  "params": {}
}
```

### 2.1 ID Rules

- Client-request IDs MAY be string or number.
- Server MUST echo the exact ID type/value in the matching response.
- Notifications MUST NOT include `id`.

## 3) Canonical Naming and Compatibility

### 3.1 Canonical Outbound Rule

Server **MUST always emit camelCase** for:
- method names (e.g., `conversation.create`)
- notification names (e.g., `tool.callRequested`)
- all object keys in `result`, `error.data`, and notification `params`

### 3.2 Inbound Compatibility Rule

Server **MUST accept** either camelCase or snake_case for known input keys and known method aliases.

Example accepted request variants:
- `maxTokens` and `max_tokens`
- `toolCallId` and `tool_call_id`

### 3.3 Collision Rule

If both camelCase and snake_case versions of the same semantic field are present in one object with different values, server MUST reject with:
- code: `1001` (`INVALID_PARAMS`)
- data.reason: `conflicting aliases`

### 3.4 Normalization Timing

Inbound normalization MUST happen before validation and business logic.

## 4) Data Types

Common structures:

```json
{
  "capabilities": {
    "streaming": true,
    "tools": true,
    "attachments": false
  },
  "model": {
    "id": "provider/model-name",
    "displayName": "Model Name"
  },
  "usage": {
    "inputTokens": 123,
    "outputTokens": 456,
    "totalTokens": 579
  }
}
```

### 4.1 Message Content Part

```json
{
  "type": "text",
  "text": "hello"
}
```

Future-safe: unknown `type` values MUST be ignored only when explicitly declared skippable by capability; otherwise fail with `UNSUPPORTED_OPERATION`.

## 5) Methods (Request/Response)

## 5.1 `session.start`

Purpose: negotiate protocol version and capabilities.

Request params:

```json
{
  "protocolVersion": "1.0",
  "client": {
    "name": "my-client",
    "version": "0.1.0"
  },
  "capabilities": {
    "streaming": true,
    "tools": true,
    "attachments": false
  }
}
```

Result:

```json
{
  "sessionId": "sess_abc123",
  "protocolVersion": "1.0",
  "server": {
    "name": "claude-app-server",
    "version": "0.1.0"
  },
  "capabilities": {
    "streaming": true,
    "tools": true,
    "attachments": false
  }
}
```

## 5.2 `session.end`

Purpose: gracefully terminate an active session.

Request params:

```json
{
  "sessionId": "sess_abc123",
  "reason": "client_shutdown"
}
```

Result:

```json
{
  "ended": true
}
```

## 5.3 `conversation.create`

Purpose: create a new conversation context.

Request params:

```json
{
  "sessionId": "sess_abc123",
  "metadata": {
    "title": "Support thread"
  }
}
```

Result:

```json
{
  "conversationId": "conv_001"
}
```

## 5.4 `conversation.send`

Purpose: submit user content; may stream updates via notifications.

Request params:

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "input": [
    { "type": "text", "text": "Explain capability negotiation." }
  ],
  "options": {
    "model": "provider/model-name",
    "maxTokens": 1024,
    "temperature": 0.2,
    "stream": true
  }
}
```

Result (non-streaming):

```json
{
  "messageId": "msg_002",
  "output": [
    { "type": "text", "text": "..." }
  ],
  "usage": {
    "inputTokens": 32,
    "outputTokens": 120,
    "totalTokens": 152
  },
  "stopReason": "end_turn"
}
```

If `options.stream=true`, immediate result MAY be:

```json
{
  "accepted": true,
  "requestId": "run_789"
}
```

and output is delivered via notifications in §6.

## 5.5 `tool.result`

Purpose: return tool execution result for a previously requested tool call.

Request params:

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "toolCallId": "tc_123",
  "output": {
    "ok": true,
    "content": [
      { "type": "text", "text": "Tool output" }
    ]
  }
}
```

Result:

```json
{
  "accepted": true
}
```

## 5.6 `conversation.cancel`

Purpose: cancel an in-flight generation.

Request params:

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "requestId": "run_789"
}
```

Result:

```json
{
  "cancelled": true
}
```

## 6) Notifications (Server → Client)

## 6.1 `conversation.started`

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "requestId": "run_789"
}
```

## 6.2 `conversation.delta`

Incremental text/content chunk.

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "requestId": "run_789",
  "delta": {
    "type": "text",
    "text": "partial text"
  }
}
```

## 6.3 `tool.callRequested`

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "requestId": "run_789",
  "toolCall": {
    "toolCallId": "tc_123",
    "name": "search",
    "arguments": {
      "query": "json-rpc"
    }
  }
}
```

## 6.4 `conversation.completed`

```json
{
  "sessionId": "sess_abc123",
  "conversationId": "conv_001",
  "requestId": "run_789",

```

```json
{

  "conversationId": "conv_001",
  "requestId": "run_789",
  "error": {
    "code": 2001,
    "message": "Upstream timeout"
  }
}
```

## 7) Error Taxonomy

All errors return JSON-RPC `error` object with numeric `code`, string `message`, optional structured `data`.

### 7.1 Categories

- `1xxx` Client/input/protocol
- `2xxx` Runtime/dependency/provider
- `3xxx` Auth/quota/policy
- `9xxx` Internal unexpected

### 7.2 Canonical Codes

| Code | Name | Meaning |
|---:|---|---|
| 1000 | INVALID_REQUEST | JSON-RPC envelope malformed |
| 1001 | INVALID_PARAMS | Params fail schema/validation |
| 1002 | METHOD_NOT_FOUND | Unknown method |
| 1003 | VERSION_UNSUPPORTED | Protocol version not supported |
| 1004 | CAPABILITY_MISMATCH | Required capability not negotiated |
| 1005 | STATE_CONFLICT | Invalid lifecycle state (e.g., ended session) |
| 2000 | UPSTREAM_UNAVAILABLE | Provider/network unavailable |
| 2001 | UPSTREAM_TIMEOUT | Provider timed out |
| 2002 | UPSTREAM_BAD_RESPONSE | Provider response invalid |
| 3000 | UNAUTHORIZED | Missing/invalid auth |
| 3001 | FORBIDDEN | Authenticated but disallowed |
| 3002 | RATE_LIMITED | Quota or rate-limited |
| 3003 | POLICY_BLOCKED | Safety/policy blocked |
| 9000 | INTERNAL_ERROR | Unexpected server error |
| 9001 | NOT_IMPLEMENTED | Declared but unimplemented feature |

### 7.3 Error Data Shape (recommended)

```json
{
  "retryable": false,
  "path": "params.options.maxTokens",
  "details": {},
  "upstream": {
    "provider": "example",
    "status": 504,
    "requestId": "abc"
  }
}
```

## 8) Capability Negotiation

Negotiation occurs in `session.start`.

Rules:
1. Client advertises desired capabilities.
2. Server returns effective capabilities.
3. A method requiring unsupported capability MUST fail with `1004 CAPABILITY_MISMATCH`.
4. Capabilities are immutable for a session after `session.start`.

Recommended capability keys:
- `streaming` (bool)
- `tools` (bool)
- `attachments` (bool)
- `jsonMode` (bool)
- `reasoningTrace` (bool; if false, trace fields omitted)

---

## 9) Compatibility Policy

## 9.1 Input Compatibility Window

Server SHOULD support snake_case aliases for one major protocol line (v1.x) to ease migration.

## 9.2 Output Stability

Server MUST NOT emit snake_case in any v1 response or notification.

## 9.3 Deprecation Signaling

When snake_case input is received, server MAY emit a warning notification (out of band) or log telemetry. It MUST still process request if otherwise valid.

---

## 10) Conformance Requirements

An implementation is v1-conformant if it:
- Implements required methods/notifications listed in this document.
- Emits canonical camelCase only.
- Accepts equivalent snake_case input aliases.
- Uses standardized error codes for listed conditions.
- Implements capability negotiation semantics.

---

## 11) Reference Validation Checklist

- [ ] Reject malformed JSON-RPC envelope with `1000`.
- [ ] Reject unknown method with `1002`.
- [ ] Normalize snake_case input before validation.
- [ ] Reject alias conflicts with `1001`.
- [ ] Emit only camelCase keys and method names.
- [ ] Enforce capability gates with `1004`.
- [ ] Emit lifecycle notifications in valid order.
- [ ] Include usage in completed output when available.
